Background: Why is this? timesync_prerequisite checks using /usr/lib/whonix/whonixcheck/apt-get-update-simulate if a package manager is currently running and waits until the package manager is done. Otherwise sdwdate may run on Whonix-Gateway while Tor is being upgraded and may therefore fail (and would report this to the user). To avoid this confusion, it checks, that no package manager is currently running.

[code]## This file is part of Whonix.

Copyright © 2012 - 2014 Patrick Schleizer adrelanos@riseup.net

See the file COPYING for copying conditions.

Whonix AppArmor Profile

Workaround for: config-package-dev clashes with AppArmor profiles

https://github.com/Whonix/Whonix/issues/66

/* {
/etc/hosts.whonix r,
/etc/resolv.conf.whonix r,
}

End of Whonix AppArmor Profile

[/code]

(You know that, for everyone else…) It basically means, “all applications are allowed to read /etc/hosts.whonix and /etc/resolv.conf.whonix”. This is only Whonix specific.

Now, when looking at https://www.whonix.org/wiki/AppArmor/Tor_Browser_Bundle I see it is using.

This makes it more difficult for Debian / Ubuntu / others to use the profile as well. I think we are best of, if as many users as possible can use these profiles. “include abstractions/whonix” looks like “Whonix specific, not a general profile, moving along”. They won’t like to use the “include abstractions/whonix”, since they don’t need it. Wouldn’t they and we be better of, if they wouldn’t need to fork the profile and wouldn’t need to remove “include abstractions/whonix” ?

My idea is to remove “#include <abstractions/whonix>” from the profiles such as https://www.whonix.org/wiki/AppArmor/Tor_Browser_Bundle, to add contents of https://www.whonix.org/wiki/AppArmor/abstractions/whonix to /etc/apparmor.d/whonix. Would that be sane? What do you think?
