This is really important, unless you are really good reading the non-graphical diff.

stage is the correct term in git terminology here.

There are staged, unstanged and untracked files.

[code]# On branch master

Changes to be committed:

(use “git reset HEAD …” to unstage)

new file: a

Changes not staged for commit:

(use “git add …” to update what will be committed)

(use “git checkout – …” to discard changes in working directory)

modified: a

Untracked files:

(use “git add …” to include in what will be committed)

x

[/code]

Untracked means, never added / committed to git.

Staged means, you used “git add”. Only changes you staged will be committed.

Not staged means, you did not use “git add” yet. Change will not be committed.

So changes can be both, staged and unstaged at the same time. I guess there is a better explanation in git ebook.

I advice to use an editor such as kate which notices when someone else changes a file. You can test that yourself. Open a text file in your editor. Then use a different editor and make changes to that file. That is what git would do. Then check if your original editor notices the change. If not, that’s bad. Can lead to situations such as this one.

Got some more improvements for this.

When there is something that can potentially go wrong (at least in theory), using local in the same line as the command is not something I like. I mean something like this.

This is because local always returns 0 (success) even if the command (in this case: “$(( $message_length -3 ))”) failed. So please define the local variables first before using them. Something like this.

local remaining_chars
remaining_chars="$(( $message_length -3 ))"

You can also define multiple local variables in one line.

local remaining_chars message_length message_without_p

And use them later.
