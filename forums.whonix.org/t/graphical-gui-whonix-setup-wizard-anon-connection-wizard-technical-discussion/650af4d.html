Let me explain the code logic in ft_m_1.

  1. DisableNetwork 0 has been in commented in /etc/tor/torrc. In other words, Tor networking has been enabled.

  2. service tor status : exit code: zero, if running, otherwise non-zero. Reason: check if the Tor daemon is running. In most cases it should be already running. Except in theoretical corner cases, where the user has run ‚Äúsudo service tor stop‚Äù or in hypothetical cases, where the daemon has tied due to a bug in Tor or due to system issues (flawed RAM, bit flip, hdd failure, etc.). I think I added the ‚Äúsleep 1‚Äù for better look and feel and to give advanced users and devs better chance to read the output, to report cases were Tor is not running for some strange reason.

  3. When Tor was running, run ‚Äúservice tor reload‚Äù. Theoretically and logically it would makes sense to ‚Äúservice tor reload‚Äù rather than ‚Äúservice tor restart‚Äù. Reload is faster. Should work. When Tor Tor was not running beforehand, start it using ‚Äúsudo service tor start‚Äù.

  4. service tor status : Due to a bug in Tor/Tor‚Äôs init script, Tor might not be running. I assume it died after ‚Äúreload‚Äù.

  5. If it was not started, run ‚Äúservice tor start‚Äù. Now Tor should really be running - reload bug or not - or something would be very wrong. Otherwise, if already started, do nothing.

  6. Working around another bug in Tor. When starting Tor fails, it does not return a non-zero return code. Run ‚Äúservice tor status‚Äù and check its exit code a final time. When success, continue as usual. When failed, notify the user.

[hr]

Wondering why not just ‚Äúservice tor restart‚Äù to avoid all the reload bug?

  • Some day I wanted to be able to actually report this bug or hope it fixes itself over time because someone else succeeds explaining it to TPO.
  • Reload, if it were working, would be faster.
  • Reload is the more appropriate way, I think.
  • Had ports to other linux distributions in mind. Reload might work there.
  • Assume the user though there is some issue with Tor, while Tor is already enabled (DisableNetwork 0). Because it didn‚Äôt serve connections yet. In that case by not using restart in all cases I wanted to prevent user‚Äôs ISP‚Äôs from seeing a connection drop/restart.
