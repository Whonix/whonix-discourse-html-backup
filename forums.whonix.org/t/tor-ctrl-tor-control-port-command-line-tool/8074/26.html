nyxnor February 4, 2022, 11:51am #25

latest script tor-ctrl-onion helps:

  • creating ephemeral onion services, with all flags possible (DiscardPK, AuthV3, MaxStreamsCloseCircuit etc)
  • deleting those services and also listing their ids.
  • adding client-side v3 onion authentication credentials, as well as removing them or viewing them

creating a service has two detach methods, from detaching to not detaching.

  • If not detaching, script will hang and the service will be active until user press Ctrl+C.
  • if detaching, flag Detach will be used and the service will be active until tor is halted, terminated, quit.

opened 02:41PM - 03 Feb 22 UTC

- ONION_CLIENT_AUTH_ADD - https://gitweb.torproject.org/torspec.git/tree/control…-spec.txt#n1990 - ONION_CLIENT_AUTH_REMOVE - https://gitweb.torproject.org/torspec.git/tree/control-spec.txt#n2033 - ONION_CLIENT_AUTH_VIEW - https://gitweb.torproject.org/torspec.git/tree/control-spec.txt#n2051 It is not the same key format used inside `ClientOnionAuthDir/file.auth_private`, as it is in base32 and the controller requires it to be in base 64. Read [TPO Client-Auth](https://community.torproject.org/onion-services/advanced/client-auth/) To generate a key, you need `openssl` (not libressl as it does not contains alg x25519) and `basez`, as it contains base32 and bas64 scripts to encode and decode. ## Create certificate: ```sh openssl genpkey -algorithm x25519 -out /tmp/k1.prv.pem ``` ## Generate base32 keys: **private key:** ```sh grep -v " PRIVATE KEY" /tmp/k1.prv.pem | base64pem -d | tail -c 32 | base32 | tr -d "=" > /tmp/k1.prv.key.base32 ``` **public key:** ```sh openssl pkey -in /tmp/k1.prv.pem -pubout | grep -v " PUBLIC KEY" | base64pem -d | tail -c 32 | base32 | tr -d "=" > /tmp/k1.pub.key.base32 ``` # Keys ## Generate base64 keys: **private key:** ```sh grep -v " PRIVATE KEY" /tmp/k1.prv.pem | base64pem -d | tail -c 32 | base64 | tr -d "=" > /tmp/k1.prv.key.base64 ``` **public key:** ```sh openssl pkey -in /tmp/k1.prv.pem -pubout | grep -v " PUBLIC KEY" | base64pem -d | tail -c 32 | base64 | tr -d "=" > /tmp/k1.pub.key.base64 ``` ## Convert key from base64 to base32: **private key:** ```sh base64 -d /tmp/k1.prv.key.base64 2>/dev/null | base32 | tr -d "=" > /tmp/k1.prv.key.base32 ``` **public key:** ```sh base64 -d /tmp/k1.pub.key.base64 2>/dev/null | base32 | tr -d "=" > /tmp/k1.pub.key.base32 ``` ## Convert key from base32 to base64: **private key:** ```sh printf '%s====\n' "$(cat /tmp/k1.prv.key.base32)" | base32 -d | base64 | tr -d "=" > /tmp/k1.prv.key.base64 ``` **public key:** ```sh printf '%s====\n' "$(cat /tmp/k1.pub.key.base32)" | base32 -d | base64 | tr -d "=" > /tmp/k1.pub.key.base64 ``` # Syntax `serviceId` is the `hostname` without `.onion`. `serviceId=HSAddress` ## ONION_CLIENT_AUTH_ADD `serviceId x25519:privateKeyInBase64` `Flags=Permanent` - This client's credentials should be stored in the filesystem. If this is not set, the client's credentials are ephemeral and stored in memory. ## ONION_CLIENT_AUTH_REMOVE `serviceId` ## ONION_CLIENT_AUTH_VIEW `[serviceId]` Tells the connected Tor to list all the stored client-side v3 client auth credentials for "HSAddress". If no "HSAddress" is provided, list all the stored client-side v3 client auth credentials.

the controller only accept keys in base64, while the configuration files can be in base32, so the issue above guides to convert keys from one to another (test before trusting, I’ve done by hand.)

I was thinking of using tor-ctrl SETEVENTS GUARD but figuring out exactly what to do on what events would require a lot of testing.

TPO does not document that on purpose, not only how to react to guards, but how to interact with the controller. They don’t teach you how to that for whatever reason they restrict people who has spent hours reading the code to make the interaction with the controller works. The specs control.spec.txt does not works for this because it doesn’t teach what the tor events means, it is just sent and up to you to decide.

For GUARDS reaction, there is github.com/mikeperry-tor/vanguards, but the debian package is almost 3 years old now.

Would be good to document all the guards important events.

Raspberry Pi TorBox log_check_config.py and log_check.py is an event based reaction to tor logs, it can restart tor or drop guards. The same could be done with the controller.

I was thinking, if we could aggregate enough info on tor guards, circuits events, we could take a better action towards a better connection.

BTW, on my experience, if I see guards problem for a long time that couldn’t make a connection even after restarting tor, I send the command tor-ctrl DROPGUARDS, to change guards. They seem to want to remove this option and I am not advising anyone to use it, just saying. If one day, they remove dropguards commands, it would still be possible to change guards by deleting some files on DataDirectory.

It is very nice the monitor command, the log messages are easy to read not raw, even though you can still see raw with events command.
Carml is built upon txtorcon, both by Meejah.

Unfortunately carml does not have a debian package, but txtorcon the library has, so possible that in the feature he provides pushes it to debian.org.

interesting commands:

  • monitor → beautify events
  • onion → create onion services
  • newid → newnym
