Klamm:

Operation timeout. The specified time-out period was reached according to the conditions.

Might indicate a very slow connection (for whatever reason, slow Tor entry guard or otherwise). To get around that you can try to increase the timeout. Currently only possible by manually editing with root rights.

kdesudo kwrite /usr/bin/update-torbrowser

Search for curl_download_max_time= and increase the timeout (in seconds).

entr0py April 30, 2016, 4:21pm #10
Klamm:

I increased the value from 180sec to different values.

/usr/bin/update-torbrowser has several instances of curl_download_max_time.

The instances that use a default of 180 secs are for downloading small stuff: connectivity checks, version numbers, signatures.

The instance of curl_download_max_time that you need to change is the one that defaults to 3600. This is the one related to the download of the package itself. (Or increase all of the timeouts. You can get a better idea of what is failing by running update-torbrowser in a terminal.)

I verified the Whonix images by gpg4win with different signature files downloaded from different Tor circuits.

The whole point of verifying using GPG signatures is that you don‚Äôt need to trust your counterparty (for file integrity that is). Feel free to download from piratebay, over http, or anywhere else that works for you. The important thing is that you need to verify the fingerprint of the key using an independent channel. For example, you could send encrypted email to random people on the mailing list asking for the Qubes Master Signing Key fingerprint. Unless they are all part of a giant conspiracy, you should wind up with the correct fingerprint. Using multiple Tor circuits may help with a MITM attack but it won‚Äôt help if the key hoster is compromised to begin with.
